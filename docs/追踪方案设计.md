# 以 task_id 为中心的错误定位流程设计

## 一、你要的完整流程

```
1. 告警 → 知道哪个 project 出问题（日志错误、服务失败等）
        ↓
2. 找到失败任务的 task_id
        ↓
3. 用 task_id 查出该任务的所有 trace_id（含首次执行和每次重试）
        ↓
4. 对每个 trace 做链路回溯，看：
   • 在哪个步骤/span 挂掉
   • 挂掉时的错误日志
   • 是否重试、重试几次
   • 重试后是否成功
   • 若仍失败，失败原因是什么
```

---

## 二、流程拆解与数据依赖

| 步骤 | 动作 | 数据来源 | 输出 |
|------|------|----------|------|
| 1 | 告警 | Grafana Alerting | project / service / 问题类型 |
| 2 | 找失败 task_id | Loki 日志（`level=error` + `task_id`） | 失败任务 task_id 列表 |
| 3 | task_id → trace 列表 | Tempo TraceQL `{span.task_id="xxx"}` | 该任务所有 trace（首次 + 重试） |
| 4 | 链路回溯 | 每个 trace 的 waterfall | 失败 span、耗时、调用链 |
| 5 | 错误日志 | Loki（tracesToLogs / trace_id 过滤） | 该 span 的错误日志 |
| 6 | 重试信息 | span attributes 或 多条 trace 对比 | 重试次数、是否成功、失败原因 |

---

## 三、面板设计（围绕该流程）

### 3.1 面板结构（4 个）

| 面板 | 职责 | 在本流程中的角色 |
|------|------|------------------|
| **Stack** | 可观测栈健康 | 排除 infra 问题 |
| **Business Metrics** | 业务 RPS/错误率/延迟 | 辅助判断服务异常 |
| **Logs** | 告警 + 错误日志 + 找 task_id | 步骤 1、2 |
| **Traces** | task_id → trace 列表 → 链路回溯 | 步骤 3、4、5、6 |

---

## 四、Logs 面板：告警 → 找 task_id

| 区域 | 内容 | 用途 |
|------|------|------|
| 告警列表 | Firing 告警 | 步骤 1：知道哪个 project 出问题 |
| 变量 | `project`（来自 Loki label） | 按告警缩小到对应 project |
| 错误日志列表 | `{project=~"$project"} \| json \| level="error"` | 步骤 2：从日志中看到 task_id |
| 按 task_id 聚合 | 每个 task_id 的错误数、最近错误时间 | 快速识别「失败任务」 |
| 操作 | 点击 task_id → 跳 Traces 面板，并传入 `task_id` | 进入步骤 3 |

**关键**：错误日志必须包含 `task_id`，且需支持从日志或聚合表点击 task_id 跳转到 Traces。

---

## 五、Traces 面板：task_id → trace 列表 → 链路回溯

### 5.1 整体布局

```
┌─────────────────────────────────────────────────────────────────────────┐
│ 输入：task_id（可从 Logs 点击传入，或手动输入）                              │
└─────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────────────┐
│ 该任务的所有 Trace（TraceQL: {span.task_id="$task_id"}）                  │
│ 表格：trace_id | 开始时间 | 耗时 | 状态(成功/失败) | 执行次序(首次/重试1/2...)  │
│ → 按时间排序，可看出：首次执行、重试1、重试2…                                │
└─────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼ 选择一条 trace
┌─────────────────────────────────────────────────────────────────────────┐
│ Trace 时间线（Waterfall）                                                  │
│ → 看调用链、哪个 span 失败、耗时分布                                        │
└─────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼ 点击失败 span
┌─────────────────────────────────────────────────────────────────────────┐
│ 该 Span 的错误日志（tracesToLogs）                                         │
│ → 挂掉的错误日志、错误信息、失败原因                                        │
└─────────────────────────────────────────────────────────────────────────┘
```

### 5.2 各区域说明

| 区域 | 内容 | 回答的问题 |
|------|------|------------|
| task_id 输入 | 变量 `$task_id`，支持从 Logs 传入 | 定位到具体失败任务 |
| Trace 列表 | TraceQL `{span.task_id="$task_id"}`，按时间排序 | 该任务有多少次执行（首次 + 重试） |
| Trace 时间线 | 选中 trace 的 waterfall | 在哪个步骤/span 挂掉 |
| Span 错误日志 | 点 span 后 tracesToLogs | 挂掉时的错误日志、失败原因 |
| 重试信息 | 从 trace 列表 + span attributes 推断 | 重试几次、重试后是否成功、为何仍失败 |

### 5.3 重试信息的获取方式

| 来源 | 含义 |
|------|------|
| Trace 列表 | 同一 task_id 多条 trace，按时间排序 ≈ 首次、重试1、重试2… |
| Span attributes | `retry_count`、`attempt`、`attempt_number` 等（若业务有埋点） |
| 错误日志 | 日志内容如 "retry 2 failed"、"failed after 3 retries" |
| Trace 状态 | 每条 trace 是否有 error span，可区分成功/失败 |

## 六、端到端流程示意

```
┌─────────────────────────────────────────────────────────────────────────────┐
│ 1. 告警：project=draw 日志错误率过高 / 服务错误率过高                          │
└─────────────────────────────────────────────────────────────────────────────┘
                                        │
                                        ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│ 2. Logs：project=draw，看错误日志，得到 task_id（如 task_abc123）              │
│    或：按 task_id 聚合，看哪些 task 错误最多                                    │
└─────────────────────────────────────────────────────────────────────────────┘
                                        │
                                        ▼ 点击 task_id
┌─────────────────────────────────────────────────────────────────────────────┐
│ 3. Traces：task_id=task_abc123                                                │
│    Trace 列表：                                                               │
│    • trace_001  10:00:01  2.3s  失败  ← 首次执行                              │
│    • trace_002  10:00:05  1.1s  失败  ← 重试1                                  │
│    • trace_003  10:00:08  0.5s  失败  ← 重试2                                  │
└─────────────────────────────────────────────────────────────────────────────┘
                                        │
                                        ▼ 选 trace_002 看 waterfall
┌─────────────────────────────────────────────────────────────────────────────┐
│ 4. Trace 时间线：发现 span "call-downstream" 标红/error                        │
│    点击该 span → 看该 span 的 Loki 日志                                       │
└─────────────────────────────────────────────────────────────────────────────┘
                                        │
                                        ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│ 5. 错误日志：Connection refused to downstream:8080                            │
│    结论：重试 2 次仍失败，原因是下游服务不可用                                  │
└─────────────────────────────────────────────────────────────────────────────┘
```

## 七、数据规范要求（需满足）

| 项目 | 要求 |
|------|------|
| 日志 | 错误日志必须包含 `task_id`、`trace_id`（来自 OTel 上下文） |
| Tempo | 所有 span 必须带 `task_id`（resource 或 span attribute） |
| 重试 | 每次重试生成新 trace_id，但共用同一 task_id |
| 错误信息 | 错误日志中需有可读的失败原因（如 error message、error_code） |

---

## 八、面板职责小结

| 面板 | 在本流程中的职责 |
|------|------------------|
| **Logs** | ① 告警入口 ② 按 project 过滤错误日志 ③ 展示/聚合 task_id ④ 点击 task_id 跳 Traces |
| **Traces** | ① 输入 task_id ② 展示该任务所有 trace（含重试） ③ 展示 trace waterfall ④ 点 span 查错误日志 ⑤ 支持判断重试次数与成败 |
| **Business Metrics** | 辅助发现服务异常 |
| **Stack** | 排除 infra 问题 |

整体设计围绕：**告警 → 找 task_id → 用 task_id 查所有 trace → 在 trace 上做链路回溯 → 定位失败 span 和错误日志 → 分析重试与失败原因**。
