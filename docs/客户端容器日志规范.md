
# 客户端容器日志规范（Promtail 采集 → Loki）

## 本规范管什么

**只管日志**：业务进程输出到 stdout/stderr 的 JSON 日志，由 Promtail 采集后写入 Loki。与 Prometheus（指标）、Tempo（追踪）是**三套独立系统**，客户端各自实现，互不依赖。

## 目标

确保 Promtail 能稳定采集日志，并支持按 `trace_id / task_id / request_id / service` 做链路排障。

> 重要前提：Promtail 只会推送**日志内容本身为 JSON 字符串**的行；Loki 最终只保留一个 label：`project`。

## 与其他系统的关系

| 系统 | 与 Loki 的关系 | 客户端需知 |
|------|----------------|------------|
| **Prometheus（指标）** | **无直接关系**。指标和日志是不同数据，收集方式不同。 | 本规范不涉及 Prometheus。若需指标，见《客户端普罗米修斯规范》。 |
| **Tempo（追踪）** | **有直接关系**。通过 `trace_id` 实现 Trace↔Log 互跳。 | 若要做 Trace↔Log 互跳，链路追踪日志必须含 `trace_id`（来自 OTel 上下文）。见《客户端 Tempo 数据规范》。 |

> **日志 vs 指标**：`code` 仅用于 Prometheus 的 HTTP 状态码 label；**日志里禁止用 `code`**，用 `status`（ok/failed/error）。

## 职责边界

- **客户端**：输出 JSON 日志，必须含 `service` 和 `project`。
- **服务端（Promtail）**：从 JSON 提取 `project` 写入 Loki label；`project` 为空则丢弃，**禁止**派生。

### Promtail 采集范围（部署侧）

Promtail 通过四个过滤规则决定采集哪些容器，由部署侧在 `.env` 中配置：

| 参数 | 说明 |
|------|------|
| `PROMTAIL_CONTAINER_NAMES` | 容器名列表（逗号分隔，精确匹配） |
| `PROMTAIL_CONTAINER_NAME_PATTERN` | 容器名正则 |
| `PROMTAIL_COMPOSE_PROJECT` | Docker Compose 项目名 |
| `PROMTAIL_COMPOSE_SERVICE` | Docker Compose 服务名（逗号分隔） |

**过滤逻辑**：
- **OR 关系**：容器匹配**任一**规则即采集，一个都不匹配则禁止采集
- **全空禁止**：四个参数全部为空时，禁止采集所有容器
- 客户端需确保：容器名 / Compose 项目 / Compose 服务符合部署侧配置，否则日志不会被采集

## 强制约定（必须遵守）

### 1）日志必须是一行一个 JSON 对象

- 每行日志输出必须是一个完整 JSON 对象（不要多行 JSON）
- 任何非 JSON 行都会被 Promtail 丢弃

### 1.1）禁止接管/修改 Root Logger（FastAPI/Uvicorn 默认日志必须保持原样）

**绝对禁止**为了“统一 JSON 日志”而清空/接管 root logger，或替换框架默认 logging 配置（尤其是 Uvicorn/FastAPI 的 access 日志格式）。

- **禁止行为**（示例）：
  - 清空 root handlers、把 root 的 formatter 改成 JSON
  - `logging.basicConfig(...)` 覆盖全局格式
  - 改写 `uvicorn` / `uvicorn.access` / `uvicorn.error` 的 handler/formatter 使其变成 JSON
- **允许行为**：
  - 业务代码使用**独立的应用 logger**（例如 `app`/`biz`），仅给该 logger 配置 JSON formatter
  - 框架日志维持默认格式（它们通常是非 JSON，会被 Promtail 丢弃，从而避免收集大量无意义日志）

### 2）必须包含字段 `service` 和 `project`

- `service`：标识服务，与 Tempo、Prometheus 命名约定保持一致（如 service=draw-gateway）。
- `project`：标识项目，**必须**在 JSON 中提供；Promtail 只提取不派生，为空则丢弃。

### 3）链路追踪相关字段（强制要求）

并非所有日志都需要做链路追踪。这里的字段要求仅针对**链路追踪日志**（例如请求入口、跨服务调用、关键业务事件等）：

**强制约束（必须遵守）**：
- ✅ `task_id` 是链路排障的**业务主键**（强制要求）
- ✅ 链路追踪日志必须输出 `trace_id`，且该 `trace_id` 必须来自 **OpenTelemetry 上下文**（或从上游 `traceparent` 提取后的上下文；禁止“随机生成伪 trace_id”）
- ✅ `task_id` 与 `trace_id` 在 tracing 语义上是一对多关系：一个 `task_id` 可对应多个 `trace_id`（每次执行/重试一个 trace）

**链路追踪日志必须包含的字段**：
- `task_id`：任务 ID（格式建议：`task_<标识符>`，如 `task_abc123`）
- `request_id`：请求 ID（强烈建议：用于跨端/跨服务关联）
- `trace_id`：32 位十六进制字符串（来自 OTel 上下文，用于 Loki↔Tempo 互跳）

**可选字段（按能力选择）**：
- `span_id`：16 位十六进制字符串

**字段关系说明**：
- `task_id` 用于追踪任务的所有执行历史（首次执行、重试等）
- 该 `trace_id` 必须与 Tempo 中的 trace 对齐（来自 OTel context）
- 建议同时包含 `span_id`，以便精确定位到日志发生的 span

> 说明：如果某进程无法接入 OTel 或无法拿到传播上下文，那么它产生的日志不应被当作“链路追踪日志”；此时可仅记录 `request_id/task_id` 做业务关联，但将无法做到 Loki↔Tempo 的 trace 互跳。

**对于普通运行日志**：不需要包含 trace_id/task_id/span_id/request_id。

### 4）可选字段

以下字段均为可选，按需使用：

**通用建议**（多数日志可包含）：
- `ts`：时间戳（RFC3339 或毫秒时间戳均可，保持一致）
- `level`：日志级别（info/warn/error）— **建议包含**，TraceLogKit 会基于 `level=~"(?i)error"` 做错误日志量统计和告警（大小写不敏感）
- `msg`：人类可读消息
- `event`：结构化事件名（便于检索/聚合）
- `duration_ms`：耗时（如适用）
- `instance`：实例标识（如 host:port 或容器/节点名）— **建议包含**，用于钉钉等告警通知中展示「告警机器」
- `device`：设备/组件标识（如磁盘、网卡ip、服务实例等）— 用于钉钉告警通知中补充机器信息

**场景扩展**（Drilldown、任务失败率等）：
- `stage`：阶段标识（如 `started`、`processing`、`completed`）— 便于按阶段过滤、统计
- `status`：状态（如 `ok`、`failed`、`error`）— 便于状态统计和告警；**任务失败率**统计依赖此字段（失败任务 = 该 task_id 下存在 `status=failed` 的日志，按 task_id 去重）
- `error_code`：错误码（出错时附带）— 便于错误分类、聚合
- `snapshot_id`：快照 ID（如适用）— 便于与 Drilldown 等联动
- `context`：嵌套对象（best-effort）— 保留完整上下文，便于排查时查看原始数据；LogQL 可用 `context.xxx` 访问子字段

## 字段示例

**链路追踪日志示例**（必须包含 service、project、task_id、trace_id）：
```json
{"ts":"2026-02-12T12:34:56.789Z","level":"info","service":"draw-gateway","project":"draw","task_id":"task_abc123","trace_id":"0123456789abcdef0123456789abcdef","span_id":"89abcdef01234567","request_id":"req_456","event":"task.started","msg":"任务开始处理","instance":"192.168.1.10:8000","device":""}
```

**普通运行日志示例**（必须包含 service、project）：
```json
{"ts":"2026-02-12T12:34:56.789Z","level":"info","service":"draw-gateway","project":"draw","instance":"192.168.1.10:8000","device":"","event":"app.startup","msg":"应用启动完成"}
```

**ai-router 示例**（service 无 `-`，project 与 service 一致）：
```json
{"ts":"2026-02-12T12:34:56.789Z","level":"info","service":"ai-router","project":"ai-router","instance":"192.168.1.10:8000","event":"app.startup","msg":"AI 路由服务启动"}
```

**带扩展字段的链路追踪日志示例**：
```json
{"ts":"2026-02-12T12:34:56.789Z","level":"error","service":"draw-gateway","project":"draw","task_id":"task_abc123","trace_id":"0123456789abcdef0123456789abcdef","span_id":"89abcdef01234567","request_id":"req_456","event":"task.failed","msg":"任务执行失败","instance":"192.168.1.10:8000","stage":"processing","status":"error","error_code":"E_TIMEOUT","snapshot_id":"snap_xyz789","context":{"retry_count":2,"last_error":"connection timeout"}}
```

## 查询示例（Grafana / LogQL）

### 按 task_id 查任务的所有执行日志（包含所有重试，推荐）

```logql
{project=~".+"} | json | task_id="task_abc123"
```

### 按 request_id 查该请求日志（推荐）

```logql
{project="draw"} | json | request_id="req_123"
```

### 按 trace_id 查全链路日志（单次执行）

```logql
{project="draw"} | json | trace_id="0123456789abcdef0123456789abcdef"
```

**说明**：
- 用 `task_id` 查询时，返回**任务的所有执行**日志（包括首次执行、重试等）
- 用 `trace_id` 查询时，返回**单次执行**的所有日志（一条完整链路）

### 按 task_id + service 过滤

```logql
{project="draw"} | json | task_id="task_abc123" | service="draw-gateway"
```

### 统计某任务的 trace 数量（执行次数）

```logql
count(count_over_time({project="draw"} | json | task_id="task_abc123" | __error__="" [24h]) by (trace_id))
```

### 同一个 trace 内按 service 过滤

```logql
{project="draw"} | json | trace_id="..." | service="draw-gateway"
```

### 按扩展字段过滤（stage、status、error_code）

```logql
{project="draw"} | json | stage="processing" | status="error"
{project="draw"} | json | error_code="E_TIMEOUT"
{project="draw"} | json | snapshot_id="snap_xyz789"
```

**任务失败率**（TraceLogKit Logs 面板与告警）：失败任务数 / 任务数，按 `task_id` 去重；失败定义 = 该 task_id 下存在 `status=~"(?i)failed"` 的日志。告警按 **project** 聚合，失败率 > 5% 时触发。
