
# 客户端 Prometheus（Metrics）暴露规范

## 本规范管什么

**只管指标**：业务进程暴露的 Prometheus 指标（HTTP 请求数、延迟等），由 Prometheus 抓取。与 Loki（日志）、Tempo（追踪）是**三套独立系统**，客户端各自实现，互不依赖。

## 目标

让 Prometheus 能抓取业务指标，支持按 `project`、`service` 做错误率、流量、延迟、CPU、内存统计与告警。

## 与其他系统的关系

| 系统 | 与 Prometheus 的关系 | 客户端需知 |
|------|----------------------|------------|
| **Loki（日志）** | **无直接关系**。指标和日志是不同数据，收集方式不同。 | 本规范不涉及 Loki。若需日志，见《客户端容器日志规范》。 |
| **Tempo（追踪）** | **无直接关系**。指标和追踪是不同数据。 | 本规范不涉及 Tempo。若需追踪，见《客户端 Tempo 数据规范》。 |

若需在 Grafana 中按同一 project/service 维度同时查指标和日志，命名约定保持一致即可。

## 职责边界

- **客户端**：暴露 `/metrics` 时，必须提供 `project`、`service` 两个 label；**建议**暴露进程指标（CPU、内存）以便 Business Metrics 面板展示。取值由部署/配置决定（如 service=draw-gateway 则 project=draw）。
- **服务端（Prometheus）**：只负责抓取、存储，**不做** project 派生或任何转换。

## 强制约定（必须遵守）

- **必须暴露 HTTP 指标端点**：`GET /metrics`
- **必须同时暴露两类 HTTP 请求指标**：`http_server_requests_total`（按 project 聚合）、`http_requests_total`（按 service 聚合），均含 `code` label；详见下文「最小可用指标集合」。
- **必须提供静态标签（推荐方式）**：
  - `project="<project>"`
  - `service="<service>"`
- **job 与 project**：抓取 target 的 `job` 必须与 `project` 一致；**Grafana 面板禁止使用 job**，只用 project 和 service。
- **禁止高基数 label**（严禁出现在 metrics labels 中）：
  - `trace_id` / `span_id` / `request_id` / `task_id` / `user_id` / `order_id` 等
- **必须排除探针类请求**：统计 HTTP 请求指标时，**不得**将 `/health`、`/ping`、`/metrics`、`/ready`、`/live` 等探针/健康检查请求计入 `http_server_requests_total`、`http_requests_total` 等业务指标，否则会污染数据，导致错误率、延迟等统计不准确。

> 说明：trace_id 等可以通过 tracing/日志关联，不应该进入 Prometheus label，否则会导致时序爆炸。

## project/service 值规范

**由客户端在暴露 metrics 时提供**，Prometheus 不派生：

- `service`：容器/服务名，例如 `draw-gateway`
- `project`：项目名，与 service 约定一致（如 service=draw-gateway 则 project=draw）

## 与 Tracing 的关联（可选增强）

如果你使用 OpenTelemetry Metrics（或支持 exemplars 的 Prometheus client），建议：

- 在 histogram/summary 上开启 exemplars（如果你的链路系统支持），exemplar 里携带 trace_id
- 这样可以从指标直接跳到 trace（Grafana 里体验更好）

## 最小可用指标集合（必须）

**必须**同时暴露以下两类 HTTP 请求指标（二者缺一不可）：

### HTTP 请求

二者**都必须暴露**，代表不同聚合层级：

| 对比项 | `http_server_requests_total` | `http_requests_total` |
|--------|------------------------------|------------------------|
| **聚合维度** | `project`：工作流/项目维度 | `service`：容器服务维度 |
| **层级关系** | 工作流整体（项目） | 工作流中的一个节点（容器服务） |
| **状态码 label** | **统一使用 `code`**，取值三位数字：`"200"`、`"404"`、`"500"` | **统一使用 `code`**，取值三位数字：`"200"`、`"404"`、`"500"` |

- 一个项目（project）可包含多个容器服务（service），因此**两类指标都必须暴露**，TraceLogKit 才能按 project 做错误率告警、按 service 做细粒度监控。
- **两种指标均必须包含 `code` label**，用于区分 4xx/5xx 错误
- 两种指标均需**排除探针路径**：`/health`、`/ping`、`/metrics`、`/ready`、`/live` 等不应计入

- `http_server_request_duration_seconds_bucket`（histogram，可选）
  - 同样需排除上述探针路径，否则延迟统计会被探针请求干扰

### 进程指标（建议，用于 CPU/内存面板）

TraceLogKit Business Metrics 面板会展示目标项目的 CPU、内存。若需展示，**建议**在 `/metrics` 中暴露以下进程指标（多数 Prometheus 客户端库会默认暴露）：

| 指标名 | 说明 |
|--------|------|
| `process_cpu_seconds_total` | 进程累计 CPU 时间（秒），用于计算 CPU 使用率 |
| `process_resident_memory_bytes` | 进程常驻内存（RSS） |

- **Python**：`prometheus_client` 默认暴露上述指标，无需额外配置
- **Go**：`prometheus.NewProcessCollector` 或 `promauto` 的 process 收集器
- **其他语言**：查阅对应 Prometheus 客户端库的 process 指标文档

未暴露时，Business Metrics 的 CPU、内存面板将无数据。

### 任务/队列（如有）

- `task_processed_total`
- `task_duration_seconds_bucket`

## 监控目标配置（部署侧）

TraceLogKit 通过 `docker/prometheus/targets/` 目录下的 YAML 文件配置抓取目标。将你的服务加入监控时：

- 每个 target 必须提供 `project`、`service` 标签；`job` 必须与 `project` 一致。
- 示例：`labels: project: "my-project", service: "my-service", job: "my-project"`

**日志采集**：指标与日志独立采集。日志由 Promtail 负责，采集范围由 `.env` 中的 `PROMTAIL_CONTAINER_NAMES`、`PROMTAIL_CONTAINER_NAME_PATTERN`、`PROMTAIL_COMPOSE_PROJECT`、`PROMTAIL_COMPOSE_SERVICE` 四个参数控制（OR 逻辑，匹配任一即采集；全空则禁止采集）。详见《客户端容器日志规范》。
